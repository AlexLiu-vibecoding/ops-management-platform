# Harness Engineering 协作框架

> 结合 SDD（规格驱动开发）和 Harness Engineering（驾驭工程），提高 AI 与用户的沟通成功率，减少返工。

## 二、文档职责

| 文档 | 职责 | 使用时机 |
|------|------|----------|
| `VIBECODING.md` | 协作经验、用户偏好、踩坑记录 | **每次开发前必读** |
| `specs/template.md` | 功能规格模板 | 新功能开发前填写 |
| `CHECKLIST.md` | 开发检查清单 | 开发过程中对照检查 |
| `CHANGELOG.md` | 变更记录 | 每次变更后记录 |
| `DEVELOPMENT.md` | 技术规范、代码风格 | 参考文档 |

## 三、工作流程

### 3.1 标准流程

```
1. 【读取上下文】
   AI 读取 VIBECODING.md，了解协作经验和用户偏好

2. 【确认需求】
   - 简单需求：直接确认
   - 复杂需求：填写 specs/template.md，用户确认后再开发

3. 【按规格开发】
   对照 CHECKLIST.md 检查清单，逐步开发

4. 【用户验收】
   按验收模板向用户展示改动，让用户验证

5. 【更新文档】
   - 更新 CHANGELOG.md 记录变更
   - 有新经验时更新 VIBECODING.md
```

### 3.2 简化流程（小改动）

```
用户提问 → AI 确认理解 → 执行修改 → 让用户验证 → 完成
```

### 3.3 完整流程（新功能）

```
用户提需求 → AI 填写规格 → 用户确认 → AI 开发 → 用户验收 → 文档更新
```

## 四、核心约束规则

### 4.1 输入约束

| 规则 | 说明 |
|------|------|
| 结构化需求 | 复杂需求必须填写规格模板 |
| 边界声明 | 明确"不做的事"，防止范围蔓延 |
| 影响分析 | 列出可能影响的功能点 |

### 4.2 执行约束

| 规则 | 说明 |
|------|------|
| 先读后写 | 修改文件前必须先读取 |
| 检查清单 | 开发过程对照检查清单 |
| 不擅自扩展 | 只做用户明确要求的 |

### 4.3 输出约束

| 规则 | 说明 |
|------|------|
| 验收模板 | 按固定格式展示改动 |
| 用户确认 | 让用户验证，不自己认为完成 |
| 经验沉淀 | 踩坑/学到的记录到 VIBECODING.md |

## 五、预期效果

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 需求理解偏差 | 高（猜测意图） | 低（规格确认） |
| 范围失控 | 经常擅自扩展 | 边界明确 |
| 遗漏步骤 | 忘测试/忘文档 | 检查清单保障 |
| 验收模糊 | 不知道做没做完 | 验收模板确认 |
| 经验流失 | 同样的坑踩两次 | 经验沉淀复用 |
| 返工次数 | 多 | 少 |

## 六、快速开始

### 用户侧

```
1. 新功能开发：说"请先写规格"
2. 发现 AI 擅自扩展：说"回滚，只需要改 xxx"
3. 踩坑/有新偏好：说"记录到 VIBECODING"
```

### AI 侧

```
1. 每次开发前：读取 VIBECODING.md
2. 复杂需求：填写 specs/template.md
3. 开发过程中：对照 CHECKLIST.md
4. 开发完成后：按验收模板确认
5. 有新经验：更新 VIBECODING.md
```
