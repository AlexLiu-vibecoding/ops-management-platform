# 变更审批

> DDL/DML 变更审批、风险分析、回滚SQL生成、分库分表支持、变更窗口管理

---

## 基本信息

- **规格编号**: `SPEC-005`
- **功能名称**: 变更审批
- **优先级**: P0 (核心功能)
- **状态**: ✅ 已完成

---

## 一、功能概述

### 1.1 变更流程

```
提交申请 → 风险分析 → 变更窗口检查 → 审批 → 执行 → 验证
```

### 1.2 支持的变更类型

| 类型 | 说明 |
|------|------|
| MySQL DDL | CREATE/ALTER/DROP |
| MySQL DML | INSERT/UPDATE/DELETE |
| Redis 命令 | SET/DEL/EXPIRE 等 |

### 1.3 核心功能

- 变更申请提交
- 自动风险分析
- 审批流程
- 自动生成回滚 SQL
- 分库分表支持
- 变更时间窗口管理
- 紧急变更支持
- 定时执行

## 四、变更窗口管理

### 4.1 变更窗口概念

变更窗口定义了允许执行变更的时间范围，用于控制风险并确保变更在可控的时间段内执行。

### 4.2 窗口类型

| 类型 | 说明 |
|------|------|
| 日常窗口 | 定期维护时间（如每周二 22:00-24:00） |
| 临时窗口 | 一次性窗口，特定日期时间 |
| 紧急窗口 | 紧急变更专用，需要更高权限 |

### 4.3 窗口属性

| 属性 | 说明 |
|------|------|
| 名称 | 窗口名称 |
| 环境ID | 适用环境 |
| 开始时间 | 窗口开始时间 |
| 结束时间 | 窗口结束时间 |
| 重复规则 | 一次性/每日/每周/每月 |
| 最小审批人数 | 执行需要的最小审批人数 |
| 允许紧急变更 | 是否允许紧急变更 |
| 状态 | 启用/禁用 |

### 4.4 全局默认策略

系统支持全局默认变更窗口策略：

| 策略项 | 默认值 | 说明 |
|--------|--------|------|
| 默认最小审批人数 | 1 | 未设置窗口时的默认审批人数 |
| 允许紧急变更 | false | 全局是否允许紧急变更 |
| 紧急变更审批人数 | 2 | 紧急变更需要的审批人数 |

### 4.5 变更申请窗口检查

提交变更申请时，系统自动检查：

1. **窗口内检查**：变更是否在允许的窗口内执行
2. **紧急变更检查**：紧急变更是否被允许
3. **审批人数检查**：是否满足最小审批人数要求

### 4.6 变更窗口 API

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/change-windows` | GET | 窗口列表 |
| `/api/v1/change-windows` | POST | 创建窗口 |
| `/api/v1/change-windows/{id}` | PUT | 更新窗口 |
| `/api/v1/change-windows/{id}` | DELETE | 删除窗口 |
| `/api/v1/change-windows/default-strategy` | GET | 获取默认策略 |
| `/api/v1/change-windows/default-strategy` | PUT | 更新默认策略 |
| `/api/v1/change-windows/check` | POST | 检查变更是否在窗口内 |

## 五、回滚 SQL 生成

### MySQL DDL
- CREATE TABLE → DROP TABLE
- ALTER TABLE → 反向 ALTER
- DROP TABLE → 无法生成（需备份）

### MySQL DML
- INSERT → DELETE
- UPDATE → 反向 UPDATE
- DELETE → INSERT（需先查询数据）

### Redis
- SET → DEL
- DEL → 无法生成（需先备份）
- EXPIRE → PERSIST 或恢复原 TTL

| 多库选择 | 选择多个数据库 |

## 七、注意事项

- 高风险变更需要多人审批
- 生产环境变更需要在时间窗口内执行
- 执行前自动生成回滚 SQL
- 大 SQL 文件自动切换到文件存储
- 紧急变更需要更高权限审批
- 变更窗口外提交会提示用户确认
