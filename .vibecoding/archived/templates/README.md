# 开发模板体系 (Development Template System)

> OpsCenter 项目全流程开发模板 - 从模糊需求到可交付代码

---

## 📚 模板概览

本模板体系基于 2026 年 Q1 最新工程化研究，采用结构化思维 (SoT)、自我验证、一次性约束、确定性输出等最佳实践，实现 **少 token、少返工、一次生成对、效率翻倍** 的目标。

### 核心特性

✅ **全流程覆盖** - 需求分析 → 架构设计 → 代码生成 → 测试 → 验证修复
✅ **结构化思维** - 每个步骤都有清晰的输入输出和验证检查点
✅ **自我验证** - 每个阶段都有自动验证机制
✅ **确定性输出** - 统一的输出格式，避免歧义
✅ **Token 优化** - 精简输出，提高效率

## 📋 模板列表

### 1. 需求分析模板

**文件**: `01-requirement-analysis.md`
**适用阶段**: 需求收集 → 架构设计
**目标**: 模糊需求 → 精准规格
**耗时**: ~10 分钟

**核心功能**:
- 需求澄清（核心功能、用户角色、操作场景、验收标准）
- 依赖分析（数据库、API、前端、权限、菜单）
- 技术决策（架构、数据、接口、安全、性能）
- 风险评估（技术、安全、兼容性、运维）
- 验收标准定义

**输出**:
```markdown
需求分析报告:
  1. 需求概述
  2. 技术方案
  3. 实施计划
  4. 风险评估
  5. 参考资料
```

**使用场景**:
- 新功能开发
- 功能重构
- 需求变更

---

### 2. 架构设计模板

**文件**: `02-architecture-design.md`
**适用阶段**: 架构设计 → 代码生成
**目标**: 需求规格 → 技术方案
**耗时**: ~12 分钟
**前置条件**: 完成需求分析报告

**核心功能**:
- 数据库设计（表结构、索引、关系）
- API 设计（RESTful 路由、参数、响应）
- 服务层设计（业务逻辑、异常、审计）
- 前端设计（页面、组件、状态、交互）
- 安全设计（权限、加密、验证）

**输出**:
```markdown
架构设计文档:
  1. 概述
  2. 数据库设计
  3. API 设计
  4. 服务层设计
  5. 前端设计
  6. 安全设计
  7. 性能设计
  8. 部署设计
```

**使用场景**:
- 新功能架构设计
- 系统重构设计
- 技术方案评审

---

### 3. 代码生成模板

**文件**: `03-code-generation.md`
**适用阶段**: 代码生成 → 单元测试
**目标**: 架构设计 → 可运行代码
**耗时**: ~10 分钟
**前置条件**: 完成架构设计文档

**核心功能**:
- 数据模型生成（SQLAlchemy 模型、Pydantic Schema）
- API 层生成（FastAPI 路由、依赖注入、权限控制）
- 服务层生成（业务逻辑、异常处理、审计日志）
- 前端生成（Vue 3 组件、状态管理、API 封装）

**输出**:
```
backend/
  app/
    models/
      [table].py
    schemas/
      [table].py
    api/
      [resource].py
    services/
      [service].py

frontend/
  src/
    views/
      [resource]/
        index.vue
    api/
      [resource].js
```

**使用场景**:
- 新功能代码生成
- 代码重构
- 补充缺失代码

---

### 4. 测试模板

**文件**: `04-testing.md`
**适用阶段**: 代码生成 → 测试执行
**目标**: 可运行代码 → 测试用例
**耗时**: ~15 分钟
**前置条件**: 完成代码生成

**核心功能**:
- 单元测试生成（服务层测试）
- API 测试生成（接口测试）
- 集成测试生成（完整流程测试）
- 测试配置生成（fixture、pytest 配置）

**输出**:
```
tests/
  unit/
    services/
      test_[service].py
  api/
    test_[resource].py
  integration/
    test_[feature].py
  helpers/
    base_service_test.py
  conftest.py
```

**使用场景**:
- 单元测试编写
- 集成测试编写
- 测试覆盖率提升

---

### 5. 验证与修复模板

**文件**: `05-verification-fix.md`
**适用阶段**: 测试执行 → 代码修复 → 测试通过
**目标**: 测试失败 → 问题定位 → 代码修复 → 测试通过
**耗时**: ~10 分钟
**前置条件**: 完成测试执行

**核心功能**:
- 测试结果分析（失败分类、根因定位）
- 代码质量检查（风格、类型、安全、性能）
- 问题修复（修复方案、修复代码）
- 回归验证（重新测试、覆盖率检查）

**输出**:
```markdown
验证与修复报告:
  1. 测试结果分析
  2. 代码质量检查
  3. 问题修复
  4. 回归验证
  5. 交付建议
```

**使用场景**:
- 测试失败修复
- 代码质量问题修复
- 回归测试验证

## 🚀 使用指南

### 完整工作流

```
1. 需求分析 (01-requirement-analysis.md)
   ↓
2. 架构设计 (02-architecture-design.md)
   ↓
3. 代码生成 (03-code-generation.md)
   ↓
4. 测试编写 (04-testing.md)
   ↓
5. 验证修复 (05-verification-fix.md)
   ↓
6. 交付
```

### 快速开始

#### 场景 1: 新功能开发

```bash
# Step 1: 需求分析
用户需求: "添加批量导出功能"
→ 使用 01-requirement-analysis.md
→ 输出: 需求分析报告

# Step 2: 架构设计
→ 使用 02-architecture-design.md
→ 输入: 需求分析报告
→ 输出: 架构设计文档

# Step 3: 代码生成
→ 使用 03-code-generation.md
→ 输入: 架构设计文档
→ 输出: 可运行代码

# Step 4: 测试编写
→ 使用 04-testing.md
→ 输入: 生成的代码
→ 输出: 测试用例

# Step 5: 验证修复
→ 使用 05-verification-fix.md
→ 输入: 测试结果
→ 输出: 修复后的代码
```

#### 场景 2: Bug 修复

```bash
# 直接使用验证与修复模板
→ 使用 05-verification-fix.md
→ 输入: 测试失败信息
→ 输出: 问题修复方案
```

#### 场景 3: 代码重构

```bash
# Step 1: 需求分析（明确重构目标）
→ 使用 01-requirement-analysis.md

# Step 2: 架构设计（设计新架构）
→ 使用 02-architecture-design.md

# Step 3: 代码生成（生成新代码）
→ 使用 03-code-generation.md

# Step 4: 测试编写（确保测试覆盖）
→ 使用 04-testing.md

# Step 5: 验证修复（验证重构效果）
→ 使用 05-verification-fix.md
```

## 🎯 最佳实践

### 1. 严格遵守模板流程

❌ **错误做法**:
```
跳过需求分析，直接写代码
→ 需求不明确，返工率高
```

✅ **正确做法**:
```
按照模板流程，逐步推进
→ 需求明确，一次成功
```

### 2. 每个阶段都有验证

❌ **错误做法**:
```
生成代码后直接交付
→ 可能有bug，需要返工
```

✅ **正确做法**:
```
每个阶段都有验证检查点
→ 问题早发现，早解决
```

### 3. 保持输出一致性

❌ **错误做法**:
```
输出格式随意变化
→ 后续处理困难
```

✅ **正确做法**:
```
严格按照模板格式输出
→ 便于自动化处理
```

### 4. 复用已有代码

❌ **错误做法**:
```
重复编写相同代码
→ 代码重复，维护困难
```

✅ **正确做法**:
```
使用模板提供的代码模板
→ 代码统一，易于维护
```

## 🆘 常见问题

### Q1: 可以跳过某些模板吗？

**A**: 不建议。每个模板都是流程的一部分，跳过可能导致：
- 需求不明确
- 架构设计不完整
- 代码质量下降
- 测试覆盖不足

### Q2: 模板不满足需求怎么办？

**A**:
1. 优先使用标准模板
2. 根据实际需求调整
3. 记录调整原因
4. 反馈给团队

### Q3: 如何提高效率？

**A**:
1. 熟悉模板流程
2. 复用已有代码
3. 使用模板提供的代码片段
4. 保持输出一致性

### Q4: 模板支持哪些语言？

**A**: 当前支持：
- 后端: Python 3.11 + FastAPI + SQLAlchemy
- 前端: Vue 3 + Vite + Element Plus

可以扩展支持其他语言。
