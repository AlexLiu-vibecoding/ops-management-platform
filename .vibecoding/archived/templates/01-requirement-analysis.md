# 需求分析模板 (Requirement Analysis Template)

> 基于 SoT (Structured of Thought) 的需求分析，确保需求清晰、可执行、可验证。

---

## 模板元数据

```
版本: v1.0
创建日期: 2026-04
适用阶段: 需求收集 → 架构设计
目标: 模糊需求 → 精准规格
```

---

## 🎯 核心原则

1. **一次性约束** - 所有约束在开始时明确，不中途变更
2. **确定性输出** - 使用结构化格式，避免歧义
3. **自我验证** - 每个步骤都有验证检查点
4. **少 token 策略** - 只输出必要信息，避免冗余

---

## 📥 输入定义

```yaml
用户需求:
  type: string
  description: 用户的原始需求描述
  required: true

上下文信息:
  type: object
  properties:
    现有功能:
      type: string
      description: 相关的现有功能说明
    相关文件:
      type: array
      items: string
      description: 需要阅读的文件路径
    技术约束:
      type: array
      items: string
      description: 技术栈或架构约束
  required: false
```

## 🧠 结构化思维 (SoT) 流程

### Step 1: 需求澄清 (3 分钟)

```
目标: 将模糊需求转化为清晰的功能点

思考框架:
1. 识别核心动作（动词）：增加/修改/删除/查询/优化
2. 识别操作对象（名词）：用户/实例/SQL/告警/配置
3. 识别约束条件：权限/性能/安全/兼容性
4. 识别验收标准：如何判断完成

输出格式:
- 核心功能: [动词] + [对象]
- 用户角色: [角色]
- 操作场景: [场景描述]
- 验收标准: [可验证的标准]
```

**验证检查点**:
- [ ] 需求是否可以用一句话描述清楚？
- [ ] 是否明确了用户角色？
- [ ] 是否有可量化的验收标准？
- [ ] 是否有未解决的技术疑问？

---

### Step 2: 依赖分析 (2 分钟)

```
目标: 识别技术依赖和前置条件

思考框架:
1. 数据库层：需要新建表/字段/索引？
2. API 层：需要新增/修改接口？
3. 前端层：需要新增页面/组件？
4. 权限层：需要新增权限码？
5. 菜单层：需要新增菜单配置？
6. 集成层：需要调用外部服务？

输出格式:
数据库依赖:
  - [ ] 新建表: 表名 + 字段
  - [ ] 新增字段: 表名 + 字段名 + 类型
  - [ ] 新建索引: 表名 + 索引定义

API 依赖:
  - [ ] 新增接口: 路由 + 方法
  - [ ] 修改接口: 路由 + 改动点

前端依赖:
  - [ ] 新增页面: 路径
  - [ ] 新增组件: 组件名
  - [ ] 新增路由: 路由配置

权限依赖:
  - [ ] 新增权限: 权限码 + 描述

菜单依赖:
  - [ ] 新增菜单: 菜单名 + 路径 + 图标
```

**验证检查点**:
- [ ] 数据库变更是否考虑了迁移？
- [ ] API 变更是否向后兼容？
- [ ] 权限是否关联到菜单？
- [ ] 前端路由是否已配置？

---

### Step 3: 技术决策 (2 分钟)

```
目标: 确定技术实现方案

思考框架:
1. 架构层面：是否需要新增服务/模块？
2. 数据层面：使用现有模型还是新建？
3. 接口层面：使用 REST 还是 GraphQL？同步还是异步？
4. 安全层面：需要什么级别的权限控制？
5. 性能层面：需要缓存/索引/分页？

决策矩阵:
方案 1: [方案描述]
  优点: [优点1, 优点2]
  缺点: [缺点1, 缺点2]
  适用场景: [场景]
  技术难度: [低/中/高]

方案 2: [方案描述]
  ...

最终选择: [方案]
理由: [选择理由]
```

**验证检查点**:
- [ ] 方案是否符合项目架构？
- [ ] 是否考虑了性能影响？
- [ ] 是否考虑了安全风险？
- [ ] 是否有技术债务？

---

### Step 4: 风险评估 (2 分钟)

```
目标: 识别潜在风险和应对措施

风险分类:
1. 技术风险: 技术可行性、性能瓶颈
2. 安全风险: SQL注入、越权访问
3. 兼容性风险: 不同环境、不同版本
4. 运维风险: 部署、监控、回滚

风险评估:
风险点: [风险描述]
  影响: [高/中/低]
  概率: [高/中/低]
  应对措施: [具体措施]
  优先级: [P0/P1/P2/P3]
```

**验证检查点**:
- [ ] 高风险是否有应对措施？
- [ ] 是否有回滚方案？
- [ ] 是否需要灰度发布？

---

### Step 5: 验收标准 (1 分钟)

```
目标: 定义清晰的验收标准

标准分类:
1. 功能验收: 功能是否正常工作
2. 性能验收: 性能是否达标
3. 安全验收: 安全是否合规
4. 兼容性验收: 是否兼容各环境

验收清单:
功能验收:
  - [ ] 标准描述
  - [ ] 测试方法
  - [ ] 验收人

性能验收:
  - [ ] 性能指标
  - [ ] 测试工具
  - [ ] 验收人
...
```

**验证检查点**:
- [ ] 验收标准是否可量化？
- [ ] 是否有明确的测试方法？
- [ ] 是否有明确的验收人？

## 💡 最佳实践

### ✅ 推荐做法

1. **使用结构化思维**
   - 每个步骤都有明确的输入和输出
   - 使用检查点验证每一步的完成度

2. **一次性约束**
   - 在开始前明确所有约束条件
   - 不要中途变更需求

3. **确定性输出**
   - 使用 YAML/JSON 格式避免歧义
   - 每个字段都有明确的类型定义

4. **自我验证**
   - 每个步骤后都进行自我检查
   - 发现问题及时修正

### ❌ 避免做法

1. **不要跳跃思考**
   - 不要跳过验证检查点
   - 不要在需求不明确时就开始设计

2. **不要模糊描述**
   - 不要使用"可能"、"应该"等模糊词汇
   - 不要缺少验收标准

3. **不要忽视风险**
   - 不要假设一切顺利
   - 不要有风险没有应对措施

---

## 📊 Token 优化策略

1. **精简输出**
   - 只输出必要信息
   - 使用缩写和符号

2. **结构化数据**
   - 使用表格和列表
   - 避免冗长的文字描述

3. **引用复用**
   - 引用现有文档而非重复
   - 使用链接而非完整复制

## 🔄 模板使用示例

### 输入

```yaml
用户需求: "给用户添加一个批量导出数据的功能"
上下文信息:
  现有功能: "用户列表页可以查看所有用户"
  相关文件:
    - "backend/app/api/users.py"
    - "frontend/src/views/users/index.vue"
  技术约束:
    - "使用 CSV 格式导出"
    - "支持选择字段导出"
```

### 输出

```markdown
# 需求分析报告

## 1. 需求概述

### 1.1 核心功能
- 功能描述: 批量导出用户数据为 CSV 文件
- 用户角色: admin
- 操作场景: 管理员在用户列表页选择导出条件

### 1.2 验收标准
- [ ] 导出文件格式正确 (CSV)
- [ ] 导出数据完整准确
- [ ] 支持分页导出 (1000+ 条数据)

## 2. 技术方案

### 2.1 架构设计
- 新增服务: `ExportService`
- 新增 API: `POST /api/v1/users/export`
- 新增前端组件: `ExportDialog`

### 2.2 数据库设计
- 无需修改表结构
- 使用现有 User 模型

### 2.3 API 设计
- 请求: `{"filters": {...}, "fields": [...]}`
- 响应: `CSV 文件流`
- 错误处理: 导出失败返回 500

### 2.4 前端设计
- 使用 Element Plus `Export` 按钮
- 字段选择器
- 进度条显示

## 3. 实施计划

### 3.1 开发任务
- [ ] 创建 ExportService
- [ ] 创建导出 API
- [ ] 创建导出对话框组件
- [ ] 添加导出按钮到用户列表页

### 3.2 测试任务
- [ ] 单元测试: ExportService
- [ ] 集成测试: 导出 API
- [ ] E2E 测试: 完整导出流程

## 4. 风险评估

| 风险点 | 影响 | 概率 | 应对措施 | 优先级 |
|--------|------|------|----------|--------|
| 大数据量导出超时 | 高 | 中 | 使用分页查询 + 流式写入 | P1 |
| CSV 特殊字符处理 | 中 | 高 | 使用标准 CSV 库 | P2 |

## 5. 参考资料
- 相关文档: [AGENTS.md](../../AGENTS.md)
- 代码示例: `backend/app/services/export_service.py`
- 最佳实践: [CSV 导出最佳实践](https://example.com)
```
