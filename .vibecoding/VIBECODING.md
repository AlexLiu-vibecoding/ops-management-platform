# VIBECODING 协作经验

> 本文档记录 AI 与用户协作开发过程中积累的经验、偏好和注意事项。
> 
> **⚠️ 每次开发前，AI 必须先读取此文件和 AGENTS.md！**

---

## 一、核心原则

| 原则 | 说明 |
|------|------|
| **先读后做** | 开发前必须先读取 AGENTS.md 和本文档 |
| **先确认后开发** | 不确定的需求，先填规格模板确认 |
| **不擅自扩展** | 用户说什么做什么，不猜测不扩展 |
| **改完要验证** | 让用户验证效果，不自己认为完成 |
| **经验要沉淀** | 踩坑/学到的经验，记录到本文档 |

---

## 二、工作流程

### 标准流程

```
1. 【读取上下文】读取 AGENTS.md 和本文档
2. 【确认需求】简单需求直接确认，复杂需求填写规格
3. 【开发实现】对照检查清单逐步开发
4. 【用户验收】按验收模板展示改动，让用户验证
5. 【经验沉淀】有新经验时更新本文档
```

### 简化流程（小改动）

```
用户提问 → AI 确认理解 → 执行修改 → 让用户验证 → 完成
```

## 三、用户偏好

### 3.1 代码风格

| 类型 | 偏好 |
|------|------|
| Python | Ruff 格式化，行宽 120 |
| 前端 | Vue 3 组合式 API，`<script setup>` |
| 表格操作列 | 统一使用 `TableActions` 组件 |
| 筛选区域 | 使用 `el-form inline` 布局 |

### 3.2 工作习惯

- **不擅自扩展范围**：用户说改什么就改什么
- **先确认再开发**：不确定的地方先问
- **改完要验证**：让用户确认效果

### 3.3 不喜欢的事情

| 行为 | 后果 |
|------|------|
| 用户只问问题，AI 却改了一堆代码 | ❌ 必须回滚 |
| 没确认需求就开始写代码 | ❌ 容易返工 |
| 代码写完不验证就说完成 | ❌ 不信任 |
| 重复问已经确认过的问题 | ❌ 浪费 token |

## 四、开发检查清单

### 4.1 开发前

```
□ 读取 AGENTS.md 了解项目结构
□ 读取相关规格文件（specs/xxx.md）
□ 确认需求范围，不擅自扩展
□ 检查现有代码，避免重复造轮子
□ 列出将要修改的文件清单
```

### 4.2 修改时

```
□ 先 read_file 读取文件内容
□ 优先使用 edit_file 精准修改
□ 保持代码风格一致
□ 不修改无关代码
```

### 4.3 修改后

```
□ 确认改动范围与需求一致
□ 没有引入硬编码
□ 没有遗漏 import 语句
```

### 4.4 特定场景

**新增 API：**
```
□ 在 main.py 注册路由
□ 添加 Pydantic Schema
□ 前端 API 封装
```

**新增菜单：**
```
□ 后端 menu.py 添加菜单配置
□ 前端 router/index.js 添加路由
□ 创建页面组件
□ 提示用户调用 /api/v1/menu/add-missing
```

**新增数据库字段：**
```
□ 更新 SQLAlchemy 模型
□ 更新 Pydantic Schema
□ 考虑数据迁移（已有数据的默认值）
```

## 五、常见错误与避坑

| 场景 | ❌ 错误 | ✅ 正确 |
|------|---------|---------|
| 端口检测 | `lsof -i:5000` | `curl -I http://localhost:5000` |
| 文件下载 | `<a href=url download>` | `fetch + blob + createObjectURL` |
| 读取文件 | `cat file.txt` | `read_file` 工具 |
| 搜索内容 | `grep "xxx" file` | `grep_file` 工具 |
| 修改文件 | 直接 write_file 覆盖 | 先 read_file 再 edit_file |
| 需求理解 | 猜测用户意图 | 先确认，再开发 |
| 服务重启 | 每次改代码都重启 | 利用热更新 |

## 六、验收模板

开发完成后，按以下格式向用户确认：

```markdown
## 完成内容

1. 修改文件：
   - frontend/src/xxx.vue - 改动说明
   - backend/app/api/xxx.py - 改动说明

2. 验收要点：
   - [ ] 功能点1
   - [ ] 功能点2

3. 请验证：
   - 刷新页面查看 xxx 功能
   - 测试 xxx 场景
```

## 七、踩坑记录

### 7.1 路由配置

**问题**：通知管理使用嵌套路由 `/config/notification`，但只有一个子页面，与其他一级路由不一致。

**解决**：单页面使用一级路由 `/notification`，多相关页面才使用嵌套路由。

### 7.2 404 错误处理

**问题**：API 返回 404 时，前端显示错误提示，影响用户体验。

**解决**：对 404 错误特殊处理，显示空数据而非错误提示。

## 八、Token 优化技巧

```
1. 精准定位
   - 使用 grep_file 搜索关键字定位文件
   - 读取文件时指定行范围

2. 避免重复操作
   - 记住已读取过的文件内容
   - 同一文件的多次修改合并为一次 edit_file

3. 减少无效对话
   - 不确定的地方一次性问清楚
   - 避免来回确认多次
```
